### 3. Code Generation Rules

- Clean, readable, structured code লিখতে হবে
- DRY principle এবং modular design follow করতে হবে

- Hardcoded secret, URL বা credential code-এ লেখা নিষেধ
- Max file size: 300 lines → বড় হলে split করতে হবে

### 4. UI & Language Rules

- **i18n System**: `i18next` + `react-i18next` — এটাই এই project-এর confirmed, official i18n solution
- **দুটো ভাষা support করে**: বাংলা (`bn`) এবং English (`en`)
- **Default ভাষা**: বাংলা (`bn`) — user প্রথমবার খুললে বাংলায় দেখবে
- **Language switch**: user header থেকে EN/BN toggle করতে পারবে; choice `localStorage`-এ save থাকে (key: `dokan360_language`)
- **Fallback**: English key না থাকলে automatically বাংলা দেখাবে

**নতুন UI text যোগ করার বাধ্যতামূলক নিয়ম:**
1. `artifacts/dokan360/src/i18n/locales/bn.json` — বাংলা text লেখো
2. `artifacts/dokan360/src/i18n/locales/en.json` — English text লেখো (একই key)
3. Component-এ hook: `const { t } = useTranslation()`
4. ব্যবহার করো: `t('section.key')` → e.g. `t('common.save')`
5. Hardcoded বাংলা বা English string সরাসরি JSX-এ লেখা সম্পূর্ণ নিষেধ

**নিষেধ:**
- নতুন component-এ `useTranslation()` ছাড়া text লেখা
- JSON file-এ key না রেখে সরাসরি string ব্যবহার
- অন্য কোনো i18n library (next-intl, react-intl ইত্যাদি) install করা
- দুটো JSON file-এ key mismatch রাখা

**Currency ও Date:**
- Currency: ৳ (Bangladeshi Taka), thousand separator সহ — e.g. ৳১,২৩৪.৫০
- Date format: DD/MM/YYYY

**UI Quality:**
- UI অবশ্যই clean, professional, SaaS-grade হতে হবে
- Bangladeshi shop owner-দের জন্য সহজবোধ্য UX নিশ্চিত করতে হবে

### 5. Project Architecture Rules — Feature Development Flow

নতুন feature বানানোর সঠিক ক্রম:

1. API contract define করো → `lib/api-spec/openapi.yaml`
2. Codegen run করো → `pnpm --filter @workspace/api-spec run codegen`
3. DB schema update করো → `lib/db/src/schema/`
4. Schema push করো → `pnpm --filter @workspace/db run push`
5. Backend module বানাও → `artifacts/api-server/src/modules/`
6. Frontend UI বানাও → `artifacts/dokan360/src/`
7. TypeScript check করো → `pnpm run typecheck`

---

### 6. Database Rules

- শুধু Supabase PostgreSQL ব্যবহার করতে হবে
- শুধু Drizzle ORM ব্যবহার করতে হবে
- Raw SQL strictly নিষেধ
- Numeric/Money column: PostgreSQL `numeric` type ব্যবহার করতে হবে; code-এ `Number()` দিয়ে convert করতে হবে
- Agent থেকে সরাসরি table delete/truncate করা নিষেধ
- Dev: `push`, Production: `migrate`

---

### 7. Auth Rules

- Frontend:
  - `supabase.auth.signInWithPassword`
  - `onAuthStateChange`
- Backend:
  - সব protected route-এ `requireAuth` middleware বাধ্যতামূলক
- Token verify: `supabaseAdmin.auth.getUser(token)` — অন্য পদ্ধতি নিষেধ
- Role system: `requireRole('admin' | 'manager' | 'cashier')`
- Session token কখনো `localStorage`-এ রাখা যাবে না — Supabase client নিজে manage করে

### 8. API Rules

- সব route `/api` দিয়ে শুরু হতে হবে
- Response format: শুধু JSON
- Validation: Zod দিয়ে request body validate করতে হবে
- Proper HTTP status code ব্যবহার করতে হবে: `200`, `201`, `400`, `401`, `403`, `404`, `500`
- Public API-তে rate limiting active রাখতে হবে

### 10. Port & Workflow Rules

| Workflow Name | Port | Purpose |
|---|---|---|
| `Backend` | `8080` | Express API server |
| `Start application` | `5000` | Vite frontend (webview / preview pane) |

- `Start application` থেকে সরাসরি `8080` call করা নিষেধ — `/api` proxy ব্যবহার করতে হবে

**NEVER START এই দুটো workflow — সম্পূর্ণ নিষেধ:**
| Workflow | কারণ |
|---|---|
| `artifacts/api-server: API Server` | `PORT=8080` ছাড়া চলে → port conflict করবে `Backend`-এর সাথে |
| `artifacts/dokan360: web` | `Start application`-এর exact duplicate → port 5000 conflict করবে |

**Workflow start করার সঠিক ক্রম:**
1. `Backend` আগে: `PORT=8080 pnpm --filter @workspace/api-server run dev`
2. তারপর `Start application`: `PORT=5000 BASE_PATH=/ pnpm --filter @workspace/dokan360 run dev`

**Crash হলে restart order:** `Backend` → `Start application`

**⚠️ Artifact Workflow Conflict:**
Replit স্বয়ংক্রিয়ভাবে `artifacts/dokan360: web` চালু করে এবং এটি port 5000 দখল করে। এই কারণে `Start application` fail করে।
- সমাধান: Replit UI-তে `artifacts/dokan360: web` manually **Stop** করার পরে `Start application` চালাতে হবে।
- Agent নিজে artifact workflow delete বা reconfigure করতে পারবে না (Replit-এর restriction)।

### 11. System Update Rule — Agent-driven Automatic

**BLUEPRINT.md হলো এই project-এর live system architecture contract — documentation নয়।**

**কখন update করতে হবে (trigger list):**
- নতুন feature add হলে
- Database schema change হলে
- API endpoint add/change হলে
- Architecture change হলে
- Port বা workflow change হলে
- Auth logic change হলে
- Frontend বা backend structure change হলে

**কীভাবে কাজ করে:**
```
User task দেয় → Agent কাজ করে → Task শেষে BLUEPRINT.md update → Change Log-এ entry যোগ
```

**BLUEPRINT.md update করার নিয়ম:**
- শুধু সংশ্লিষ্ট section update করতে হবে — পুরো file rewrite নয়
- পুরনো critical architecture মুছে ফেলা নিষেধ — শুধু replace করো
- Hallucinate করা নিষেধ — actual codebase দেখে লিখতে হবে
- সবসময় Change Log-এ নতুন entry যোগ করতে হবে
- Backward compatibility info সবসময় preserve করতে হবে

**Change Log entry format:**
```
### YYYY-MM-DD — [কী হয়েছে সংক্ষেপে]
- **Changed**: কী বদলেছে
- **Why**: কেন বদলেছে
- **Impacted**: কোন module/file affected
```

### 15. Forbidden Actions — এগুলো কখনো করা যাবে না

- `any` type ব্যবহার

- Raw SQL

- `npm` বা `yarn` ব্যবহার

- Missing error handling
- `pnpm-lock.yaml` manually edit
